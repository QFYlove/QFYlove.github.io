目前的学习目标&进度（持续更新中）

学习计划

战略建议

  1. 首选赛道锁定： 将**游戏测试开发（SDET）**作为第一优先级，**技术策划（Technical Designer）**作为第二优先级。这两个岗位能最大化利用 CS 专业的代码基础，同时规避了高强度的算法考核。
  2. 学习策略重构： 放弃“纯开发”学习路线，采用**“产品化开发”**（Product-Oriented Development）学习法。即在编写代码时，不再关注算法复杂度，而是关注业务逻辑的实现与用户体验的数据化埋点。
  3. 寒假执行方案： 利用寒假集中突击 Python 自动化框架（Appium/Airtest）与 Unity 基础逻辑，产出具备“测试+开发+产品”三位一体属性的复合型项目作品，而非传统的图书管理系统。

技能树

自动化框架（核心竞争力）

  • Appium / Selenium: 这是目前移动端测试的行业标准。需要理解如何通过代码控制手机模拟器，模拟用户的点击、滑动操作。
  • Airtest: 由网易开发的跨平台 UI 自动化测试框架，在游戏行业（尤其是国内大厂）应用极广。掌握 Airtest 对于申请网易系的实习岗位具有极高的加分效应。
  • 技能转化点： 学习这些框架并不需要深奥的算法，只需要掌握 Python 的基础语法、类与对象的调用以及 API 的阅读能力。

脚本语言（Python）

  • 学习重点： 文件 I/O 操作（读取日志）、正则表达式（分析日志）、网络请求库（Requests）、多线程（并发测试）。
  • 避坑指南： 不需要去刷 Python 的算法题，而是要多写脚本。例如：“编写一个脚本，自动遍历游戏中的所有 UI 界面并截图”。

操作系统与环境

  • Android ADB 指令： 理解如何通过命令行安装应用、抓取日志（logcat）、监控内存与 CPU。
  • Linux 基础： 服务器端的测试往往需要登录 Linux 后台查看服务状态，掌握基本的 Shell 指令是必修课。

融入产品思维的“开发学习法”

逆向工程分析法

  • 案例： 拆解《原神》的角色切换机制。

  • 技术视角（Dev）： 思考代码层面如何实现？（例如：对象池管理、状态机切换、输入缓冲）。

  • 产品视角（PM）： 思考为什么要这样设计？（例如：通过元素反应机制鼓励多角色抽取，从而提升卡池流水；通过切换冷却时间（CD）防止玩家无限连招，保持战斗节奏）。

  • 产出物： 在 GitHub 项目中，不仅上传代码，还要附带一份 Markdown 格式的《机制分析文档》（PRD）。在文档中详细阐述：“如果我是策划，我为什么制定这个数值？如果我是程序，我如何低成本实现它？”

数据驱动的开发

  • 实践： 在写一个小游戏 Demo 时，不要只写玩法逻辑，要加上“数据埋点”代码。

  • 代码示例： 当玩家死亡时，不只是重置游戏，而是编写一个函数 LogDeathEvent(Vector3 position, int timePlayed)，将死亡坐标和存活时间记录到 CSV 文件中。

  • 简历亮点： “开发了一款 Demo，并自行设计了数据采集模块，通过分析 100 次试玩的死亡热力图，优化了关卡难度曲线。” —— 这句话对面试官的杀伤力极大，它证明了你不仅是 Coder，更是 Product Owner。

行动指南

工具与基础构建

**目标：**掌握 Python 自动化基础与 Unity 核心逻辑，摒弃算法刷题。

学习重点 具体动作与资源 产品思维融入点
Python 自动化入门 学习 Python 基础语法（列表、字典、循环）。安装 Appium 与 Android 模拟器。完成“自动启动 APP 并点击按钮”的脚本。 思考：如何设计脚本才能适应不同分辨率的手机？（兼容性思维）
测试框架进阶 深入学习 Airtest 框架。尝试对一个开源小游戏编写自动化回归测试脚本（如：自动跑图 5 分钟）。 思考：什么样的 Bug 是致命的（崩溃），什么样的 Bug 是体验类的？（优先级思维）
Unity 与 C#基础 跟着 Unity 官方教程完成“Ruby’s Adventure”或类似 2D 项目。理解生命周期（Start/Update）与碰撞检测。 思考：Unity 的组件化设计（Component）如何降低了开发耦合度？（架构思维）

项目实战与差异化打造

**目标：**产出一个“多维展示”的作品集（Portfolio）。不要做一个大而全的 RPG，要做一个“深而精”的机制 Demo。

  • 项目选题： “具备自动化测试与数据分析功能的游戏原型”。
  • 开发内容：
    1. 玩法层： 一个简单的点击类或跑酷类游戏（利用免费素材网站如 Kenney 获取美术资源）。
    2. 测试层： 编写一套 Python 脚本，能够自动运行这个游戏，并检测帧率（FPS）是否稳定。
    3. 产品层： 在游戏中加入参数调整 UI（例如：动态调整角色移动速度），并撰写一份文档，分析不同速度对游戏手感的影响。
  • 成果形态： 一个 GitHub 仓库，包含：游戏源码 + 自动化测试脚本 + 产品分析文档（PDF）。

笔试与面试应对

  • 笔试：
    • 测试岗： 笔试题通常包含基础逻辑、测试用例设计（例如：“如何测试一个微信发红包功能”）、计算机网络基础（TCP/UDP）、以及少量的简单编程题（通常不涉及复杂算法）。
    • 应对： 刷《牛客网》的“测试工程师”题库，而非“算法工程师”题库。
  • 面试环节：
    • 面试官心理： 面试测试或技术策划的面试官，通常更看重候选人的沟通逻辑和项目落地能力。
    • 当被问及“你为什么不做开发而做测试/策划”时，不要说“因为我不喜欢算法”（这是减分项），而要说：“因为我更喜欢站在全局视角去保障产品的最终交付质量。开发关注的是代码的局部实现，而测试/策划关注的是用户体验的最终闭环。我的 CS 背景让我能比纯策划更懂技术边界，比纯测试更懂代码逻辑。” —— 这是一个满分回答，它将你的劣势（不喜欢纯算法）包装成了你的差异化优势（全局观、产品观）。
